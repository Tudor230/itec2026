# Project Architecture

This document explains how the project is structured, how authentication and routing fit together, and how contributors should extend the codebase without fighting the existing setup.

## Goals

- Keep everything inside a single TanStack Start application.
- Use file-based routing with TanStack Router.
- Use Auth0 for authentication without a separate backend service.
- Keep shared logic small and centralized.
- Make common contribution paths predictable.

## Stack

- Framework: TanStack Start with React
- Routing: TanStack Router file-based routes
- Auth: Auth0 React SDK
- Data layer: TanStack Query
- Build tool: Vite
- Styling: Tailwind CSS v4 plus project CSS tokens in `src/styles.css`
- Language: TypeScript

## High-Level Design

At runtime, the app is composed in this order:

1. `src/router.tsx` creates the application router from the generated route tree.
2. `src/routes/__root.tsx` defines the HTML shell and mounts global providers.
3. `TanStackQueryProvider` provides a shared `QueryClient`.
4. `AuthProvider` mounts the Auth0 provider when env configuration is present.
5. Routes under `src/routes` render inside the shared shell.

The project is intentionally frontend-heavy. Authentication, role checks, and return-to behavior all happen in the app itself.

## Directory Responsibilities

### `src/auth`

Authentication wiring and route protection live here.

- `AuthProvider.tsx`: mounts `Auth0Provider`, handles post-login redirect, and exposes whether auth is configured.
- `ProtectedRoute.tsx`: protects authenticated pages and optionally enforces required roles.

Put new auth-specific wrappers, hooks, or session utilities here.

### `src/components`

Reusable UI components.

- `Header.tsx` and `Footer.tsx` are shared site chrome.
- `ThemeToggle.tsx` owns theme switching.
- `components/auth/*` contains auth-specific UI such as login, logout, setup notice, and user display.

If a component is visual and reused across routes, it belongs here.

### `src/integrations`

Wrappers around external libraries.

- `tanstack-query/root-provider.tsx`: creates and provides the shared query client.
- `tanstack-query/devtools.tsx`: query devtools integration.

If a new library needs a provider or shared adapter, prefer adding it here rather than scattering setup code across routes.

### `src/lib`

Small, reusable utilities and configuration helpers.

- `auth0-config.ts`: parses Auth0 env values and builds redirect targets.
- `auth-claims.ts`: extracts roles from Auth0 user claims and evaluates role requirements.
- `utils.ts`: generic utility helpers.

Keep this directory focused on logic with low UI coupling.

### `src/routes`

Application pages and route tree entries.

- `__root.tsx`: root shell route.
- `index.tsx`: public landing page.
- `about.tsx`: implementation notes and rationale.
- `dashboard.tsx`: authenticated page example.
- `demo/*`: starter/demo routes.

Each file maps to a route. Prefer keeping route files thin and moving reusable logic into `src/components`, `src/auth`, or `src/lib`.

### Generated Files

- `src/routeTree.gen.ts` is generated by TanStack Router tooling.

Do not hand-edit generated files. Change the route files in `src/routes` instead.

## Routing Model

The router is created in `src/router.tsx` using TanStack Router and the generated route tree.

Important characteristics:

- Routing is file-based.
- Shared shell behavior belongs in `src/routes/__root.tsx`.
- New pages are typically added by creating a new route file in `src/routes`.
- Route protection is handled by wrapping route content with `ProtectedRoute`.

Example pattern:

```tsx
import { createFileRoute } from '@tanstack/react-router'
import ProtectedRoute from '../auth/ProtectedRoute'

export const Route = createFileRoute('/settings')({
  component: Settings,
})

function Settings() {
  return (
    <ProtectedRoute>
      <main>Protected content</main>
    </ProtectedRoute>
  )
}
```

## Authentication Model

The app uses the Auth0 React SDK directly in the frontend.

### Auth Flow

1. `AuthProvider.tsx` reads Auth0 configuration from Vite env vars.
2. If config is present, it mounts `Auth0Provider`.
3. Login buttons call `loginWithRedirect`.
4. The current route is stored in `appState.returnTo`.
5. After Auth0 redirects back, `onRedirectCallback` sends the browser back to the intended route.

### Environment Variables

Current auth configuration is based on these variables:

- `VITE_AUTH0_DOMAIN`
- `VITE_AUTH0_CLIENT_ID`
- `VITE_AUTH0_REDIRECT_URI`
- `VITE_AUTH0_LOGOUT_RETURN_TO`
- `VITE_AUTH0_ROLES_CLAIM` optional

Contributors should remember that Vite only reads `.env` at startup. After changing env vars, restart the dev server.

### Role Guards

Role checks are implemented in `src/lib/auth-claims.ts` and enforced by `src/auth/ProtectedRoute.tsx`.

Current behavior:

- If `VITE_AUTH0_ROLES_CLAIM` is set, the app checks that exact claim key first.
- Otherwise it falls back to:
  - `user.roles`
  - any claim ending with `/roles`
  - any claim ending with `:roles`

Preferred Auth0 Action pattern:

```js
exports.onExecutePostLogin = async (event, api) => {
  const namespace = 'https://your-app.example.com/'
  const roles = event.authorization?.roles || []

  api.idToken.setCustomClaim(`${namespace}roles`, roles)
}
```

Example route requiring a role:

```tsx
<ProtectedRoute requiredRoles={['admin']}>
  <AdminPage />
</ProtectedRoute>
```

## Data Fetching Model

The project already includes TanStack Query. Use it when you need client-side caching, background refetching, or async state management.

Guidelines:

- Shared query client setup belongs in `src/integrations/tanstack-query`.
- Route components should avoid creating their own query client instances.
- Prefer colocating query hooks near the feature they support when the app grows.

## Styling Model

Styling is split between:

- `src/styles.css` for global tokens, theme variables, animation helpers, and layout primitives
- Tailwind utility classes inside components and routes

Contributors should:

- reuse existing CSS variables before inventing new colors or shadows
- keep visual changes consistent with the current glassy lagoon-style theme
- avoid large one-off style blocks inside route files unless the styles are truly route-specific

## Contribution Guide

### Before You Change Code

1. Read the nearest related file before editing.
2. Keep changes local to the feature you are touching.
3. Avoid broad refactors unless they are required by the task.

### Local Workflow

Install and run the app:

```bash
npm install
npm run dev
```

Useful validation commands:

```bash
npm run build
npm run lint
npm run test
```

If you change formatting-sensitive files, run:

```bash
npm run check
```

### Adding a New Route

1. Create a file in `src/routes`.
2. Export a TanStack Router route via `createFileRoute`.
3. If the page requires auth, wrap it in `ProtectedRoute`.
4. If the page requires roles, pass `requiredRoles` and optionally `match`.
5. Add navigation links only if the route should be user-visible.

### Adding a Protected or Role-Based Page

Use these patterns:

Authenticated only:

```tsx
<ProtectedRoute>
  <Page />
</ProtectedRoute>
```

Require a role:

```tsx
<ProtectedRoute requiredRoles={['admin']}>
  <Page />
</ProtectedRoute>
```

Require any of several roles:

```tsx
<ProtectedRoute requiredRoles={['admin', 'editor']} match="any">
  <Page />
</ProtectedRoute>
```

### When to Add to `src/lib`

Add code to `src/lib` when it is:

- reused by multiple routes or components
- not tied to rendering
- not provider setup

If it owns React context or a provider, it likely belongs in `src/auth` or `src/integrations` instead.

### When to Add to `components/auth`

Put code there when it is UI that depends on the auth session, for example:

- login buttons
- logout buttons
- profile cards
- setup notices

Keep raw auth logic out of these components when possible. Prefer using helpers from `src/lib` and wrappers from `src/auth`.

## Conventions

- Prefer small focused files over large multi-purpose components.
- Keep route files readable and move reusable logic out quickly.
- Use environment variables for deployment-specific settings.
- Keep Auth0 claim names explicit and namespaced.
- Prefer extending existing helpers before duplicating auth logic.
- Do not hand-edit generated files.

## Common Contribution Tasks

### Add a New Public Page

- Create a route file under `src/routes`
- Add a link in `Header.tsx` only if appropriate
- Reuse existing page shells and styling tokens

### Add an Admin Page

- Create a route file
- Wrap the page in `ProtectedRoute requiredRoles={['admin']}`
- Confirm the Auth0 Action emits the expected roles claim
- Optionally set `VITE_AUTH0_ROLES_CLAIM` if the claim key is custom

### Change Auth0 Configuration

- Update `.env`
- Restart `npm run dev`
- Re-check callback URL, logout URL, and allowed web origins in Auth0

### Troubleshoot Missing Roles

If a protected page says the user lacks access:

1. Confirm the Auth0 Action writes roles into the ID token.
2. Confirm the claim name matches `VITE_AUTH0_ROLES_CLAIM` if that env var is set.
3. Check the user profile display in the app to see whether roles are being resolved.
4. Restart the dev server after changing env values.

## Suggested Next Improvements

These are good follow-up contributions if the project expands:

- add a dedicated admin route example
- add a 403 unauthorized page instead of an inline message
- add a protected layout route for authenticated sections
- add tests around claim parsing and route guards

## Contribution Checklist

Before considering a change complete:

1. Does it follow the existing directory boundaries?
2. Did you avoid editing generated files manually?
3. Did you run the relevant validation command?
4. If auth behavior changed, did you test login, logout, and route protection?
5. If env vars changed, did you restart the dev server?
